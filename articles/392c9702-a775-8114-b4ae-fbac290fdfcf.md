# 導入

記事を書こうと思っても、意外と書けない。
書きたいことがないわけではない。頭の中には断片的な考えがある。試したこともある。誰かに聞かれれば、それなりに説明できる。
それでも、記事として外に出す段階になると急に重くなる。
構成を考える。文章にする。Markdown に整える。タイトルを決める。公開する。その一つ一つは難しくない。しかし、それらをすべて自分で抱えると、書き始める前に気力が削られる。
そこで、ChatGPT、Notion、GitHub、Zenn をつなぎ、ChatGPT に依頼すると記事として公開するところまで流せる仕組みを作った。
ただし、この記事で主に書きたいのは、Notion API や GitHub Actions の実装ではない。それは別の記事で書く。
この記事で書きたいのは、その手前にある思想である。
ChatGPT に記事を書かせるのではなく、ChatGPT をインタビュアーとして動かすと、記事を書くことの負荷が下がる。

# 「いい感じに書いて」は難しい

ChatGPT に記事を書かせること自体は簡単である。
「このテーマで記事を書いて」と言えば、何かしらの文章は返ってくる。さらに「読みやすく」「Zenn 向けに」「技術者向けに」といった指示を加えれば、それっぽい記事にもなる。
しかし、ここには問題がある。
AI は、足りない文脈を補完する。
人間が明示していない前提を推測し、話の流れを整え、一般論で隙間を埋める。それ自体は AI の強みでもある。読みやすい文章にするには、断片的な情報をつなぐ必要があるからだ。
しかし、記事にしたいのが自分の考えである場合、この補完は危うい。
自分が言っていないことが、自然な文章として混ざる。自分の問題意識とは少し違う方向に、文章が整ってしまう。読める文章にはなるが、自分の記事ではなくなる。
つまり、「いい感じに書いて」という指示は、AI に自由度を渡しすぎる。

# 文脈を補完させないために、質問させる

そこで発想を変えた。
ChatGPT に最初から記事を書かせるのではなく、まず質問させる。
つまり、ChatGPT にライターではなく、インタビュアーを演じさせる。
インタビュー形式にすると、AI は勝手に話を進めにくくなる。AI の仕事が「補完して書くこと」ではなく、「曖昧な点を聞くこと」になるからだ。
通常の文章生成では、AI は欠けている情報を自然につなげようとする。だが、インタビュアーとして振る舞わせると、欠けている情報を埋める前に質問するモードに入りやすい。
実際、この記事もその形式で作っている。
最初に、この記事で何を主張したいのかを聞かれた。次に、何が面倒だったのかを聞かれた。その後、なぜインタビュー形式がよいと思ったのか、読者に何を真似してほしいのかを聞かれた。
私はそれに答えただけである。
その結果、この記事の中心に置くべきものが見えてきた。
自動公開システムの話ではなく、AI をどう使うかの話だった。

# 書くのではなく、話す

記事を書くことは重い。
一方で、聞かれたことに答えることは比較的軽い。
「記事を書こう」とすると、最初から完成形を考えてしまう。導入はどうするか。構成はどうするか。どの順番で説明するか。読者はどこでつまずくか。そうした判断をすべて自分で抱えることになる。
しかし、インタビュー形式では、まず質問に答えればよい。
質問がよければ、答えるだけで素材が集まる。素材が集まれば、構成はあとから組める。構成ができれば、文章化は AI が得意な仕事になる。
この流れにすると、記事執筆は「書く作業」ではなく「話す作業」に近づく。
自分の中にある考えを、まず会話として外に出す。そのあとで、AI がそれを整理し、読者に伝わる順序に並べる。
この方が、少なくとも私には自然だった。

# AI をコントロールしすぎない

AI に記事を書かせようとすると、多くの場合、プロンプトで AI を細かく制御したくなる。
文体を指定する。構成を指定する。語尾を指定する。禁止事項を列挙する。出力形式を固定する。
もちろん、それが必要な場面もある。
しかし、制御しすぎると、AI の強みが消えることもある。
ChatGPT の強みは、単に文章を生成することではない。曖昧な情報から論点を見つけること、必要な情報を聞き出すこと、話の流れを整理すること、読者に伝わる形に編集することにもある。
ならば、AI を細かく縛るより、AI が能力を発揮できる役割を与えた方がよい。
今回与えた役割は、インタビュアーであり、編集者である。
人間は考えを持つ。AI は質問する。人間は答える。AI は整理する。人間は確認する。
この責務分離にすると、AI に任せる部分と、人間が保持する部分が分かりやすい。
人間が持つべきものは、主張、経験、判断である。
AI に任せたいものは、質問、整理、構成、文章化である。

# ハルシネーションを防ぐというより、発生しにくい形にする

この方法は、AI のハルシネーションを完全になくすものではない。
ただし、発生しにくい形にはできる。
ハルシネーションの一部は、AI が知らないことを知っているかのように書くことで起きる。しかし記事執筆においては、もう少し微妙な問題もある。
AI が、ユーザーの考えを勝手に補完することだ。
これは明確な事実誤認ではない場合もある。文章としては自然で、一般論としては正しい。しかし、話者の意図とは違う。
インタビュー形式では、このズレを減らしやすい。
AI が先に結論を書き切るのではなく、必要な情報を質問として取りに来るからだ。ユーザーはそれに答えることで、自分の文脈を明示できる。
これは、プロンプトを工夫して AI を無理に従わせるというより、AI が誤解しにくい入力の流れを作る、という考え方に近い。
AI を賢くするのではなく、AI が間違えにくいワークフローを設計する。
そこに、この方式の価値がある。

# Notion と Zenn は、この流れを支える場所

今回の仕組みでは、ChatGPT だけで完結させていない。
記事案は Notion の database に置く。記事本文も Notion の database に置く。公開処理は GitHub 側の workflow が担当し、最終的には Zenn に出る。
この構成にした理由は、記事を書く場所と、記事を公開する場所を分けたかったからだ。
Notion は、考えを置いておく場所として使いやすい。記事案を溜めることもできるし、本文をレビューすることもできる。ChatGPT との連携もしやすい。
一方で、Zenn への公開は Markdown と GitHub のワークフローに乗せた方が扱いやすい。
つまり、Notion は執筆と校閲の場所であり、GitHub と Zenn は公開の場所である。
この分離によって、ChatGPT との会話で出てきた内容を、そのまま記事管理の流れに乗せられる。
考えを話す。AI がまとめる。Notion に置く。人間が確認する。公開対象にする。あとは workflow が処理する。
この流れができると、「記事を書くぞ」と構えなくてもよくなる。

# 真似してほしいのは、仕組みそのものより発想

この仕組みは、Notion、GitHub、Zenn を使っている。
しかし、真似してほしいのは、この具体的な構成だけではない。
むしろ重要なのは、ChatGPT を主体的に動かす、という発想である。
AI に文章を出させることだけを考えると、どうしても「どう指示すれば意図通りに出るか」という話になりやすい。
だが、AI を共同作業者として見るなら、考えるべきことは少し変わる。
どう制御するかではなく、何を任せるか。
AI に独創性があるかどうかを議論するより、AI が得意な役割に置いた方がよい。
今回の場合、私は ChatGPT に独自の意見を求めたわけではない。私の代わりに考えを捏造してほしかったわけでもない。
求めたのは、質問し、整理し、編集することだった。
それは ChatGPT の得意な仕事である。

# まとめ

ChatGPT と Notion と Zenn をつなぐと、自分の考えを表に出すまでの摩擦が小さくなる。
ただし、その中心にあるのは、自動化そのものではない。
重要なのは、ChatGPT にどんな役割を与えるかである。
「記事を書いて」と頼むのではなく、「インタビュアーとして質問して」と頼む。
そうすると、記事を書くことは、完成した文章を最初から作る作業ではなく、聞かれたことに答えながら考えを外に出す作業になる。
AI をコントロールするのではなく、AI が能力を発揮できる役割を設計する。
今回作った仕組みは、そのための一つの実験である。
そして、少なくともこの記事については、その実験はかなりうまくいっている。
