# TL;DR

ChatGPT にいきなり記事を書かせると、足りない文脈を AI が自然に補完してしまいます。
自分の考えを記事にするには、ChatGPT をライターではなくインタビュアーとして使う方が安全です。
現在の運用では、AI 明記 callout の直後に TL;DR を置き、先に要点を固定してから本文へ入ります。
この方式は ChatGPT の中で完結するため、Codex の token を使いません。Codex を使ってまで記事を書きたくない人や、ChatGPT を細く長く使いたい人にも向いています。
人間は考えを持ち、AI は質問し、整理し、編集します。記事を書く作業を、聞かれたことに答える作業へ変えるのがこの記事の中心です。

# 導入

記事を書こうと思っても、意外と書けない。
書きたいことはある。試したこともある。誰かに聞かれれば説明できる。それでも、記事として外に出す段階になると急に重くなる。
構成を考える。文章にする。Markdown に整える。タイトルを決める。公開する。どれも単体では難しくないが、全部を自分で抱えると、書き始める前に気力が削られる。
そこで、ChatGPT、Notion、GitHub、Zenn をつなぎ、考えを記事として公開するところまで流せる仕組みを作った。
この記事で扱うのは実装ではない。ChatGPT に記事を書かせる前に、どのような役割を与えるべきか、という話である。

# 問題: AIに任せると文脈が補完される

ChatGPT に「このテーマで記事を書いて」と頼めば、それらしい文章は返ってくる。
しかし、そこには危うさがある。AI は足りない文脈を補完する。明示されていない前提を推測し、一般論で隙間を埋め、自然な文章に整える。
これは AI の強みでもある。ただし、自分の考えを記事にしたい場合は問題になる。
自分が言っていないことが、自然な文章として混ざる。主張が少しずれる。読める文章にはなるが、自分の記事ではなくなる。
「いい感じに書いて」は、AI に自由度を渡しすぎる。

# 方針: 書かせる前に質問させる

そこで、ChatGPT に最初から記事を書かせるのをやめた。
代わりに、インタビュアーとして質問させることにした。
インタビュー形式にすると、AI の仕事は「足りない情報を補完して書くこと」ではなく、「足りない情報を聞くこと」になる。勝手に話を進めるより、曖昧な点を確認する方向に動きやすい。
この記事もその形式で作っている。
最初に、この記事で何を主張したいのかを聞かれた。次に、何が面倒だったのかを聞かれた。その後、なぜインタビュー形式がよいのか、読者に何を真似してほしいのかを聞かれた。
私はそれに答えただけである。
その結果、この記事の中心が見えてきた。これは自動公開システムの話である前に、AI にどのような役割を与えるかの話だった。

# 効果: 書く作業を話す作業に変える

記事を書くことは重い。一方で、聞かれたことに答えることは比較的軽い。
「記事を書こう」とすると、最初から完成形を考えてしまう。導入、構成、説明順、読者のつまずきどころを、すべて自分で抱えることになる。
インタビュー形式では、まず質問に答えればよい。質問がよければ素材が集まる。素材が集まれば構成は後から組める。構成ができれば、文章化は AI が得意な仕事になる。
この流れにすると、記事執筆は「書く作業」ではなく「話す作業」に近づく。

# 設計思想: AIを細かく縛らない

AI に記事を書かせようとすると、プロンプトで細かく制御したくなる。
文体、構成、語尾、禁止事項、出力形式を指定する。必要な場面もあるが、制御しすぎると AI の強みが消える。
ChatGPT の強みは、文章生成だけではない。曖昧な情報から論点を見つけること、必要な情報を聞き出すこと、話の流れを整理すること、読者に伝わる形に編集することにもある。
だから、細かく縛るより、能力を発揮できる役割を与える。
今回の役割は、インタビュアーであり、編集者である。
人間は考えを持つ。AI は質問する。人間は答える。AI は整理する。人間は確認する。
この責務分離にすると、AI に任せる部分と、人間が保持する部分が分かりやすい。

# ハルシネーションを起きにくくする

この方法は、ハルシネーションを完全になくすものではない。
ただし、起きにくい形にはできる。
記事執筆で問題になるのは、明確な事実誤認だけではない。AI がユーザーの考えを勝手に補完することも問題になる。文章として自然で、一般論として正しくても、話者の意図とは違うことがある。
インタビュー形式では、AI が先に結論を書き切らない。必要な情報を質問として取りに来る。ユーザーはそれに答えることで、自分の文脈を明示できる。
AI を賢くするのではなく、AI が間違えにくい入力の流れを作る。
そこに、この方式の価値がある。

# Notion と Zenn は流れを支える場所

今回の仕組みでは、ChatGPT だけで完結させていない。
記事案と記事本文は Notion に置く。公開処理は GitHub 側の workflow が担当し、最終的には Zenn に出る。
Notion は、考えを置き、レビューする場所として使いやすい。一方で、Zenn への公開は Markdown と GitHub のワークフローに乗せた方が扱いやすい。
つまり、Notion は執筆と校閲の場所であり、GitHub と Zenn は公開の場所である。
考えを話す。AI がまとめる。Notion に置く。確認する。公開対象にする。あとは workflow が処理する。
この流れができると、「記事を書くぞ」と構えなくてもよくなる。

# つまり何が言いたいか

重要なのは、Notion、GitHub、Zenn という具体的な構成そのものではない。
ChatGPT を主体的に動かすという発想である。
AI に文章を出させることだけを考えると、「どう指示すれば意図通りに出るか」という話になりやすい。
しかし、AI を共同作業者として見るなら、考えるべきことは変わる。
どう制御するかではなく、何を任せるか。
今回、私は ChatGPT に独自の意見を求めたわけではない。私の代わりに考えを捏造してほしかったわけでもない。
求めたのは、質問し、整理し、編集することだった。

# ChatGPT だけで完結させる

この運用は ChatGPT の中で閉じている。
記事案を出す。質問する。構成を作る。本文を書く。推敲する。Notion に整理する。ここまでは ChatGPT でできる。Codex を使って repository を直接編集したり、コード生成の枠で記事本文を書いたりする必要はない。
これは小さい違いに見えるが、運用上は重要である。Codex の token を記事執筆で消費しない。Codex を使ってまで記事を書く気がない人でも使える。ChatGPT を token 制限の中で細々と使っている人にも向いている。
また、ChatGPT と Codex では再現性の性質が違う。ChatGPT は会話の文脈をもとに、質問、整理、推敲を続けるのに向いている。一方で、Codex は repository 上の実装作業に向いている。記事本文は ChatGPT に閉じ、同期や変換の実装は必要なときだけ別の仕組みに任せる。この分離が、無理のない運用になる。

# 現在の運用では TL;DR を先に置く

この記事を書いた後、AI 共同執筆記事では冒頭の形を少し変えました。
現在は、AI による執筆であることを明記する callout の直後に `# TL;DR` を置きます。本文を読む前に、記事の結論、読者が得るもの、重要な注意点を短く示すためです。
これは、AI 生成文が丁寧になりすぎる問題への対策でもあります。導入で背景を説明する前に、まず要点を固定します。そのうえで、導入では「なぜこの話をするのか」だけを書くようにします。

# まとめ

ChatGPT と Notion と Zenn をつなぐと、自分の考えを表に出すまでの摩擦が小さくなる。
ただし、中心にあるのは自動化そのものではない。重要なのは、ChatGPT にどんな役割を与えるかである。
「記事を書いて」と頼むのではなく、「インタビュアーとして質問して」と頼む。
それだけで、記事を書くことは、完成した文章を最初から作る作業ではなく、聞かれたことに答えながら考えを外に出す作業になる。
AI をコントロールするのではなく、AI が能力を発揮できる役割を設計する。
今回作った仕組みは、そのための一つの実験である。少なくともこの記事については、その実験はうまく機能している。
