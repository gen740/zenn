# TL;DR

Nix は、最初から NixOS や package 作成まで理解しようとすると難しく見えます。
最初は `flake.nix`、`flake.lock`、`nix develop` だけ見れば十分です。
Nix を「環境を固定する道具」として使うと、プロジェクトごとに同じ開発環境へ入れます。
この記事では、Python の devShell を例にして、Nix の入口だけを説明します。

# 導入

Nix は難しく見えます。
実際、Nix の周辺には多くの概念があります。Nix language、nixpkgs、NixOS、home-manager、flake、overlay、module system、package 作成。最初に全部を見ようとすると、何から理解すればよいのか分からなくなります。
しかし、最初から全部を理解する必要はありません。
この記事では、Nix を「環境を固定する道具」として扱います。NixOS の話はしません。package 作成の細かい話もしません。まず、プロジェクトごとに同じ開発環境へ入るところだけを見ます。

# まず NixOS から入らない

Nix と聞くと、NixOS を思い浮かべる人がいます。
NixOS は重要です。ただし、Nix 入門の入口としては重いです。OS 全体を Nix で管理する話に入ると、system configuration、service、user 管理、boot loader なども同時に出てきます。
開発環境を再現したいだけなら、そこまで必要ありません。
まずは通常の Linux や macOS の上で Nix を使えばよいです。プロジェクトの repository に `flake.nix` を置き、`nix develop` で shell に入ります。それだけで、プロジェクトごとの環境を作れます。

# Nix は何を固定するのか

Nix を package manager と呼ぶことはできます。
しかし、入門では package manager として見るより、環境を固定する仕組みとして見た方が分かりやすいです。
通常の開発環境では、必要な tool を手で入れます。

```bash
brew install python
brew install pkg-config
brew install cmake
```

この方法は手軽です。ただし、どの version を使っているか、後から分かりにくくなります。別の machine で同じ環境を作るときも、手順と現在の package repository の状態に依存します。
Nix では、必要な tool を `flake.nix` に書きます。

```nix
{
  description = "minimal Python dev shell";

  inputs = {
    nixpkgs.url = "github:NixOS/nixpkgs/nixos-unstable";
  };

  outputs = { self, nixpkgs }:
    let
      system = "x86_64-linux";
      pkgs = import nixpkgs { inherit system; };
    in
    {
      devShells.${system}.default = pkgs.mkShell {
        packages = [
          pkgs.python312
          pkgs.ruff
        ];
      };
    };
}
```

この repository で次を実行します。

```bash
nix develop
```

すると、`python` と `ruff` が使える shell に入ります。

```bash
python --version
ruff --version
```

重要なのは、tool を global に入れているわけではないことです。この project のための shell に入っています。

# flake.nix は環境の入口である

`flake.nix` は、この project が何に依存しているかを書く file です。
先ほどの例では、`nixpkgs` を input として使っています。`nixpkgs` は、Nix package の巨大な集合です。Python、Ruby、Node.js、C compiler、LaTeX、CMake など、多くの package がここから来ます。
`outputs` では、この flake が何を提供するかを書きます。
今回提供しているのは `devShells.${system}.default` です。これは、`nix develop` で入る default shell です。
つまり、最小構成では次の対応だけを覚えればよいです。

```plain text
flake.nix
  ↓
devShells.<system>.default
  ↓
nix develop
  ↓
project 用 shell
```

# flake.lock が入力を固定する

`nix develop` を実行すると、`flake.lock` が作られます。
`flake.lock` は、input の具体的な revision を固定する file です。`flake.nix` に `nixpkgs.url = "github:NixOS/nixpkgs/nixos-unstable";` と書いていても、実際に使った commit は `flake.lock` に記録されます。
これにより、別の machine でも同じ revision の nixpkgs を使えます。

```plain text
flake.nix  は依存関係の指定を書く
flake.lock は実際に使う revision を固定する
```

再現性を考えるなら、`flake.lock` は repository に commit します。
`flake.lock` を更新したいときは、明示的に更新します。

```bash
nix flake update
```

この操作をすると、依存する nixpkgs などの input が新しい revision に更新されます。逆に言えば、勝手には変わりません。

# devShell は project ごとの作業部屋である

`devShell` は、project ごとの作業環境です。
たとえば、ある project では Python 3.12 と Ruff を使う。別の project では Python 3.11 と CMake を使う。さらに別の project では LaTeX と Poppler を使う。このような違いを、project ごとの `flake.nix` に書けます。
global environment を汚さず、project ごとの shell に入ります。

```bash
cd project-a
nix develop
```

別の project では、別の shell に入ります。

```bash
cd project-b
nix develop
```

これにより、machine 全体の環境ではなく、project に必要な環境を管理できます。

# shellHook で初期化もできる

package を並べるだけで足りない場合もあります。
たとえば、環境変数を設定したい場合があります。そのときは `shellHook` を使えます。

```nix
{
  description = "dev shell with environment variable";

  inputs = {
    nixpkgs.url = "github:NixOS/nixpkgs/nixos-unstable";
  };

  outputs = { self, nixpkgs }:
    let
      system = "x86_64-linux";
      pkgs = import nixpkgs { inherit system; };
    in
    {
      devShells.${system}.default = pkgs.mkShell {
        packages = [
          pkgs.python312
        ];

        shellHook = ''
          export PROJECT_DATA=$PWD/data
          echo "PROJECT_DATA=$PROJECT_DATA"
        '';
      };
    };
}
```

`nix develop` に入ると、`PROJECT_DATA` が設定されます。

```bash
echo $PROJECT_DATA
```

科学計算や解析環境では、binary だけでは足りないことがあります。環境変数、初期化 script、database の場所なども必要になります。その場合でも、shell に入った時点で必要な初期化を済ませられます。

# Docker とは目的が少し違う

Nix と Docker は、どちらも環境の再現性に関係します。
ただし、目的は少し違います。
Docker は container image を作ります。process を隔離し、その image の中で動かします。deployment や service 実行に向いています。
Nix の devShell は、開発や解析のための shell を作ります。現在の作業 directory に入り、editor や terminal からそのまま使えます。
どちらが上という話ではありません。
開発中に必要な compiler、formatter、language runtime、解析 tool を揃えたいなら、まず devShell が便利です。service として固めて配布したいなら、Docker が合う場面もあります。

# 最初に覚える command

最初に覚える command は多くありません。

```bash
nix develop
```

project の devShell に入ります。

```bash
nix flake update
```

`flake.lock` を更新します。

```bash
nix flake show
```

flake が提供している output を確認します。
最初はこれで十分です。

# 最初に避けるもの

Nix に入門するとき、最初からすべてを理解しようとしない方がよいです。
特に、次の話題は後でよいです。

* NixOS の system configuration
* home-manager
* overlay
* package の詳細な作り方
* module system
* cross compilation
* binary cache の運用

これらは重要です。しかし、最初の一歩ではありません。
最初の一歩は、project に `flake.nix` を置き、`nix develop` で同じ shell に入れることです。

# まとめ

Nix は、最初から全部を理解しようとすると難しいです。
しかし、入口を `flake.nix` と `nix develop` に絞ると、かなり単純になります。
`flake.nix` に環境を書く。`flake.lock` で input を固定する。`nix develop` で project 用 shell に入る。
まずはこれだけでよいです。
この使い方だけでも、開発環境や解析環境を project ごとに固定できます。特に、compiler、language runtime、formatter、解析 tool、環境変数が絡む project では効果があります。
Nix は、package を入れるためだけの道具ではありません。
同じ project に戻ったとき、同じ環境に戻るための道具です。
