Neovim 0.13 の `vim.ui.img` を使って、TeX の Live Preview を作りました。左側で TeX ソースを書き、右側に PDF のプレビューを表示します。保存すると `latexmk` で再コンパイルし、プレビューを更新します。さらに `<CR><CR>` を押すと SyncTeX の forward search でカーソル位置に対応する PDF ページへ移動し、該当行を半透明の矩形でハイライトします。
この記事は、TeX プレビュー環境そのものの紹介というより、`vim.ui.img` を実用的な UI の部品として使うと何ができるのか、という実例です。

# 作ったもの

作ったものは、Neovim の中で完結する TeX Live Preview です。
機能は次の通りです。

* 右側の split に PDF のページを表示する
* `*.tex` の保存時に `latexmk` で再コンパイルする
* コンパイル後にプレビューを自動更新する
* `<CR><CR>` で SyncTeX forward search を実行する
* 対応する PDF ページへ移動する
* SyncTeX が返した位置を半透明の矩形でハイライトする
* ウィンドウのリサイズに追従して表示を調整する

実装は Lua です。PDF は直接表示せず、`pdftoppm` で PNG に変換してから `vim.ui.img` で表示しています。

# なぜ作ったのか

論文を書くにあたって、TeX の編集環境を見直していました。
それまでは Vim と Skim を組み合わせて使っていました。Skim 自体に大きな不満があったわけではありません。PDF ビューアとして十分に便利ですし、TeX のプレビュー環境としても普通に使えます。
ただ、やはり別アプリであることは少し扱いづらいです。エディタと PDF ビューアが別々に存在すると、アプリケーションを切り替える必要があります。さらに、画面上でウィンドウを並べる必要もあります。大きな問題ではありませんが、論文を書いていると何度も発生する小さな摩擦になります。
Neovim の中に閉じられるなら、その方が自然です。編集、保存、コンパイル、プレビュー、対応箇所へのジャンプが同じエコシステムの中で完結します。アプリケーションを切り替えず、ウィンドウ管理も Neovim 側の split として扱えます。
この「エディタの中に閉じる」という性質が、今回作ったものの主な動機です。

# Neovim 内で PDF を扱えるようになった

TeX の Live Preview 自体は以前から可能でした。Vim や Neovim で TeX を編集し、Skim や Zathura のような外部 PDF ビューアに表示する構成は一般的です。
難しかったのは、PDF プレビューを Neovim の中に自然に組み込むことです。Neovim は基本的に文字グリッドの UI なので、PDF を表示するには、ページを画像としてターミナル上に配置する必要があります。
Neovim 0.13 の `vim.ui.img` により、この部分を Neovim の API として扱えるようになりました。kitty graphics protocol に対応した環境であれば、画像を Neovim の UI の一部として配置できます。
今回の実装では、PDF の各ページを PNG に変換し、その画像を `vim.ui.img` で右ペインに表示しています。

# 全体構成

処理の流れは単純です。

```plain text
TeX source
  ↓ save
latexmk
  ↓
main.pdf
  ↓
pdftoppm
  ↓
PNG
  ↓
vim.ui.img
  ↓
Neovim の右ペインに表示
```

SyncTeX を使う場合は、カーソル位置から PDF 上の位置を問い合わせます。

```plain text
cursor line
  ↓
synctex view
  ↓
page number and box
  ↓
PDF coordinate to terminal cells
  ↓
vim.ui.img overlay
```

PDF 本体を直接表示するのではなく、表示対象のページだけを PNG に変換します。プレビュー用途では、必要なページを必要なタイミングで画像化できれば十分です。

# 右ペインに PDF を表示する

起動時には、現在の TeX ソースを開いているウィンドウを覚えたうえで、右側に split を作ります。

```lua
S.source_win = vim.api.nvim_get_current_win()

vim.cmd('rightbelow vsplit')
S.preview_win = vim.api.nvim_get_current_win()
S.preview_buf = vim.api.nvim_create_buf(false, true)
vim.api.nvim_win_set_buf(S.preview_win, S.preview_buf)

vim.bo[S.preview_buf].bufhidden = 'wipe'
vim.bo[S.preview_buf].buftype = 'nofile'

vim.api.nvim_set_current_win(S.source_win)
```

プレビュー用の buffer は scratch buffer です。ユーザーが編集するものではなく、画像を置くための領域として使います。
表示する PDF ページは `pdftoppm` で PNG に変換します。

```lua
vim.system({
  'pdftoppm', '-png',
  '-f', tostring(page), '-l', tostring(page),
  '-singlefile',
  '-scale-to-x', tostring(target_px),
  '-scale-to-y', '-1',
  S.pdf, S.prefix,
}, { text = false }, function(res)
  -- PNG を読み込んで vim.ui.img.set する
end)
```

変換後の PNG を読み込み、`vim.ui.img.set` に渡します。

```lua
local ok, id = pcall(vim.ui.img.set, blob, {
  row = pr,
  col = pc,
  width = w,
  height = h,
  zindex = cfg.zindex,
})
```

ここで `row` と `col` は画面上のセル位置、`width` と `height` はセル単位の表示サイズです。つまり、画像をどのウィンドウのどこに置くかを自前で計算します。

# ターミナルのセルは正方形ではない

画像をターミナル上に表示するときに問題になるのが、セルの縦横比です。
ターミナルの 1 セルは正方形ではありません。多くの場合、文字セルは横よりも縦に長くなります。そのため、画像のピクセル比率だけを見てセル数に変換すると、表示が潰れたり伸びたりします。
この実装では、端末にセルのピクセルサイズを問い合わせています。

```lua
vim.tty.request('\027[16t', { timeout = 200, on_timeout = finish }, function(resp)
  local h, w = resp:match('\027%[6;(%d+);(%d+)t')
  if h and w and tonumber(w) > 0 then
    cfg.cell_aspect = tonumber(h) / tonumber(w)
    finish()
    return true
  end
end)
```

取得したセルの縦横比を使って、PDF ページが歪まないように表示サイズを決めます。

```lua
local function fit_cells(img_w, img_h, cols, rows)
  local unit_w = img_w
  local unit_h = img_h / cfg.cell_aspect
  local scale = math.min(cols / unit_w, rows / unit_h)
  local w = math.max(1, math.floor(unit_w * scale))
  local h = math.max(1, math.floor(unit_h * scale))
  return w, h
end
```

この補正を入れないと、PDF ページの見た目と実際の配置矩形がずれます。後述する SyncTeX ハイライトも、ページ画像の位置とサイズに依存するため、セルの縦横比は重要です。

# 保存時に自動コンパイルする

TeX ファイルを保存したら自動でコンパイルします。

```lua
vim.api.nvim_create_autocmd('BufWritePost', {
  group = S.augroup,
  pattern = '*.tex',
  callback = function(args)
    local f = vim.api.nvim_buf_get_name(args.buf)
    if f:sub(1, #S.root) == S.root then
      compile()
    end
  end,
})
```

`compile` では `latexmk` を呼びます。

```lua
vim.system({ 'latexmk', '-e', [[$dvipdf=q/dvipdfmx -z0 %O -o %D %S/;]] }, {
  cwd = S.root,
  text = true,
}, function(res)
  vim.schedule(function()
    S.compiling = false
    if res.code == 0 then
      show(S.page)
    else
      set_status('✗ compile error')
      notify('latexmk failed (see log)', vim.log.levels.ERROR)
    end
  end)
end)
```

プレビュー用途では、最終成果物としての PDF のサイズよりも、保存してから表示が更新されるまでの体感が重要です。そのため、この実装では preview 用に `dvipdfmx -z0` を使っています。圧縮を切ることで、図の多い PDF でも更新の待ち時間を減らす意図です。
また、保存を連続で行ったときにコンパイルが重ならないように、`compiling` と `pending` を持っています。

```lua
if S.compiling then
  S.pending = true
  return
end
```

すでにコンパイル中なら、次のコンパイルを pending として記録します。現在のコンパイルが終わったあと、pending が立っていればもう一度コンパイルします。

# SyncTeX でカーソル位置に対応する PDF ページへ移動する

TeX のプレビューで重要なのは、ソースと PDF の対応です。
この実装では、normal mode の `<CR><CR>` に SyncTeX forward search を割り当てています。

```lua
vim.keymap.set('n', '<CR><CR>', sync_from_cursor, {
  buffer = buf,
  silent = true,
  desc = 'auto_preview: SyncTeX forward search',
})
```

カーソル位置から、行番号、列番号、ファイル名を取り出します。

```lua
local pos = vim.api.nvim_win_get_cursor(win)
local line, colnr = pos[1], pos[2] + 1
```

それを `synctex view` に渡します。

```lua
vim.system({
  'synctex', 'view',
  '-i', string.format('%d:%d:%s', line, colnr, file),
  '-o', S.pdf,
}, { text = true }, function(res)
  -- 出力から page と box を読む
end)
```

SyncTeX の出力から、ページ番号と対応する box を取り出します。

```lua
local page = out:match('Page:(%d+)')
local h = tonumber(out:match('[\r\n]h:([%d.%-]+)'))
local v = tonumber(out:match('[\r\n]v:([%d.%-]+)'))
local bw = tonumber(out:match('[\r\n]W:([%d.%-]+)'))
local bh = tonumber(out:match('[\r\n]H:([%d.%-]+)'))
```

ページ番号が分かれば、そのページを表示します。box が分かれば、表示後にその位置をハイライトします。

```lua
show(tonumber(page), box and function()
  place_highlight(box)
end or nil)
```

# ハイライトは 1x1 PNG を拡大している

この実装で一番面白い部分は、ハイライトです。
PDF 上の対応箇所を目立たせるには、ページ画像の上に半透明の矩形を重ねればよいです。しかし、Neovim のテキストハイライトや extmark は、PDF 画像の上に重ねるためのものではありません。
そこで、半透明の 1x1 PNG を用意し、それを `vim.ui.img` で拡大表示しています。

```lua
local HL_PNG = vim.base64.decode(
  'iVBORw0KGgoAAAANSUhEUgAAAAEAAAABCAYAAAAfFcSJAAAADUlEQVR42mP4f5chDwAHBgJLBoePSgAAAABJRU5ErkJggg=='
)
```

これをページ画像よりも高い `zindex` で配置します。

```lua
local ok, id = pcall(vim.ui.img.set, HL_PNG, {
  row = hl_row,
  col = hl_col,
  width = hl_w,
  height = hl_h,
  zindex = cfg.zindex + 1,
})
```

つまり、PDF ページも画像、ハイライトも画像です。画像レイヤーとして重ねることで、Neovim の中に GUI 的な見た目を作っています。
この方法なら、ハイライトの位置と大きさを PDF の座標から計算して、そのまま画像として重ねられます。`vim.ui.img` を単なる画像表示機能としてではなく、オーバーレイ UI の部品として使っている点が重要です。

# PDF 座標をターミナル座標へ変換する

SyncTeX が返すのは PDF 上の座標です。一方で `vim.ui.img.set` が要求するのは、ターミナル上のセル座標です。
そのため、PDF 座標から現在表示中のページ画像上のセル座標へ変換します。

```lua
local hl_col = g.col + math.floor((box.left / pw) * g.w)
local hl_row = g.row + math.floor((box.top / ph) * g.h)
local hl_w = math.max(1, math.ceil((box.w / pw) * g.w))
local hl_h = math.max(1, math.ceil((box.h / ph) * g.h))
```

ここで `pw` と `ph` は PDF ページの幅と高さです。`g` は現在配置しているページ画像の位置と大きさです。
この変換によって、PDF 上の box が、ターミナル上で実際に表示されているページ画像のどこに対応するかを求めています。
ページ画像が中央寄せされている場合もあるため、単純にウィンドウ左上からの相対位置では足りません。実際に配置した画像の `row`、`col`、`width`、`height` を状態として保持しておく必要があります。

# リサイズにも追従する

Neovim の split は動きます。ターミナルサイズも変わります。そのため、最初に画像を置いただけでは不十分です。
この実装では、`VimResized` と `WinResized` で現在のページを再表示します。

```lua
vim.api.nvim_create_autocmd({ 'VimResized', 'WinResized' }, {
  group = S.augroup,
  callback = function()
    vim.schedule(refresh)
  end,
})
```

リサイズ時は TeX を再コンパイルする必要はありません。PDF は変わっていないので、現在のページを新しいペインサイズに合わせて再レンダリングすれば十分です。

# 実装して分かったこと

今回の実装で分かったのは、`vim.ui.img` は単に画像を表示する API ではなく、Neovim の UI を拡張する部品として使えるということです。
PDF ページを置くだけなら、画像表示機能です。しかし、ページ画像の上に別の画像を重ね、SyncTeX の座標に合わせて動かすと、これは UI になります。
もちろん、現時点では環境依存があります。kitty graphics protocol に対応した端末が必要ですし、Neovim も `vim.ui.img` を持つビルドである必要があります。さらに、今回の例では `pdftoppm`、`synctex`、`latexmk` も必要です。
ただし、条件が揃うなら、Neovim の中で TeX の編集とプレビューをかなり自然に統合できます。
