Neovim 0.13 の `vim.ui.img` を使って、TeX の Live Preview を作りました。左側で TeX ソースを書き、右側に PDF のプレビューを表示します。保存すると `latexmk` で再コンパイルし、表示を更新します。さらに `<CR><CR>` を押すと、SyncTeX の forward search でカーソル位置に対応する PDF ページへ移動し、該当箇所を半透明の矩形でハイライトします。
この記事は、TeX 環境そのものの紹介ではありません。`vim.ui.img` を Neovim の UI 部品として使うと、どのようなことができるのかを示す実装例です。

# 作ったもの

作ったのは、Neovim の中で完結する TeX Live Preview です。
主な機能は次の通りです。

* 右側の split に PDF ページを表示する
* `*.tex` の保存時に `latexmk` で再コンパイルする
* コンパイル後にプレビューを自動更新する
* `<CR><CR>` で SyncTeX forward search を実行する
* カーソル位置に対応する PDF ページへ移動する
* 対応箇所を半透明の矩形でハイライトする
* ウィンドウのリサイズに追従して表示を調整する

実装は Lua です。PDF を直接表示するのではなく、必要なページだけを `pdftoppm` で PNG に変換し、その画像を `vim.ui.img` で表示しています。

# なぜ Neovim の中に PDF を出したかったのか

論文を書くにあたって、TeX の編集環境を見直していました。これまでは Vim と Skim を組み合わせて使っていました。Skim は PDF ビューアとして十分便利ですし、TeX のプレビュー環境としても普通に使えます。
ただし、エディタと PDF ビューアが別アプリであることには小さな摩擦があります。アプリケーションを切り替える必要があり、画面上でウィンドウも並べなければなりません。大きな問題ではありませんが、論文を書いていると何度も発生します。
Neovim の中に閉じられれば、編集、保存、コンパイル、プレビュー、対応箇所へのジャンプを同じ環境で扱えます。外部ビューアを否定したいわけではありません。Neovim の split として PDF を扱えるなら、その方が自分の作業には合っている、という判断です。

# vim.ui.img で何が変わったのか

TeX の Live Preview 自体は新しいものではありません。Vim や Neovim で TeX を編集し、Skim や Zathura のような外部 PDF ビューアに表示する構成は以前からあります。
難しかったのは、PDF プレビューを Neovim の中に自然に組み込むことです。Neovim は基本的に文字グリッドの UI です。PDF をそのまま表示するには、ページを画像としてターミナル上に配置する必要があります。
Neovim 0.13 の `vim.ui.img` により、この部分を Neovim の API として扱えるようになりました。kitty graphics protocol に対応した環境であれば、画像を Neovim の UI の一部として配置できます。
今回の実装では、PDF の各ページを PNG に変換し、その画像を `vim.ui.img` で右ペインに表示しています。

# 全体の流れ

処理の流れは単純です。

```plain text
TeX source
  ↓ save
latexmk
  ↓
main.pdf
  ↓
pdftoppm
  ↓
PNG
  ↓
vim.ui.img
  ↓
right split
```

SyncTeX を使う場合は、カーソル位置から PDF 上の対応箇所を問い合わせます。

```plain text
cursor position
  ↓
synctex view
  ↓
page and box
  ↓
convert PDF coordinates to terminal cells
  ↓
vim.ui.img overlay
```

PDF 全体を常に画像化する必要はありません。プレビューで必要なのは、今表示したいページを、必要なタイミングで画像に変換することです。

# 右ペインに PDF ページを表示する

起動時には、現在の TeX ソースを開いているウィンドウを覚えたうえで、右側に split を作ります。

```lua
S.source_win = vim.api.nvim_get_current_win()

vim.cmd('rightbelow vsplit')
S.preview_win = vim.api.nvim_get_current_win()
S.preview_buf = vim.api.nvim_create_buf(false, true)
vim.api.nvim_win_set_buf(S.preview_win, S.preview_buf)

vim.bo[S.preview_buf].bufhidden = 'wipe'
vim.bo[S.preview_buf].buftype = 'nofile'

vim.api.nvim_set_current_win(S.source_win)
```

プレビュー用 buffer は scratch buffer です。ユーザーが編集する対象ではなく、画像を配置するための領域として使います。
表示する PDF ページは `pdftoppm` で PNG に変換します。

```lua
vim.system({
  'pdftoppm', '-png',
  '-f', tostring(page), '-l', tostring(page),
  '-singlefile',
  '-scale-to-x', tostring(target_px),
  '-scale-to-y', '-1',
  S.pdf, S.prefix,
}, { text = false }, function(res)
  -- PNG を読み込んで vim.ui.img.set する
end)
```

変換後の PNG を読み込み、`vim.ui.img.set` に渡します。

```lua
local ok, id = pcall(vim.ui.img.set, blob, {
  row = pr,
  col = pc,
  width = w,
  height = h,
  zindex = cfg.zindex,
})
```

`row` と `col` は画面上のセル位置です。`width` と `height` はセル単位の表示サイズです。つまり、画像をどの位置に、どの大きさで置くかは自前で計算します。

# セルの縦横比を補正する

画像をターミナル上に表示するとき、単純にピクセル比率をセル数へ変換すると表示が歪みます。ターミナルの 1 セルは正方形ではないからです。多くの場合、文字セルは横よりも縦に長くなります。
この実装では、端末にセルのピクセルサイズを問い合わせています。

```lua
vim.tty.request('\027[16t', { timeout = 200, on_timeout = finish }, function(resp)
  local h, w = resp:match('\027%[6;(%d+);(%d+)t')
  if h and w and tonumber(w) > 0 then
    cfg.cell_aspect = tonumber(h) / tonumber(w)
    finish()
    return true
  end
end)
```

取得したセルの縦横比を使って、PDF ページが歪まないように表示サイズを決めます。

```lua
local function fit_cells(img_w, img_h, cols, rows)
  local unit_w = img_w
  local unit_h = img_h / cfg.cell_aspect
  local scale = math.min(cols / unit_w, rows / unit_h)
  local w = math.max(1, math.floor(unit_w * scale))
  local h = math.max(1, math.floor(unit_h * scale))
  return w, h
end
```

この補正は見た目だけの問題ではありません。SyncTeX のハイライトも、ページ画像の位置とサイズに依存します。表示サイズがずれると、ハイライト位置もずれます。

# 保存時に自動コンパイルする

TeX ファイルを保存したら、自動で `latexmk` を呼びます。

```lua
vim.api.nvim_create_autocmd('BufWritePost', {
  group = S.augroup,
  pattern = '*.tex',
  callback = function(args)
    local f = vim.api.nvim_buf_get_name(args.buf)
    if f:sub(1, #S.root) == S.root then
      compile()
    end
  end,
})
```

コンパイルに成功したら、現在表示しているページを再描画します。

```lua
vim.system({ 'latexmk', '-e', [[$dvipdf=q/dvipdfmx -z0 %O -o %D %S/;]] }, {
  cwd = S.root,
  text = true,
}, function(res)
  vim.schedule(function()
    S.compiling = false
    if res.code == 0 then
      show(S.page)
    else
      set_status('✗ compile error')
      notify('latexmk failed (see log)', vim.log.levels.ERROR)
    end
  end)
end)
```

プレビュー用途では、最終成果物としての PDF サイズよりも、保存してから表示が更新されるまでの体感が重要です。そのため、この実装では preview 用に `dvipdfmx -z0` を使っています。圧縮を切ることで、図の多い PDF でも更新待ちを減らす意図です。
保存を連続で行ったときにコンパイルが重ならないように、`compiling` と `pending` も持たせています。

```lua
if S.compiling then
  S.pending = true
  return
end
```

すでにコンパイル中なら、次のコンパイルを pending として記録します。現在のコンパイルが終わったあと、pending が立っていればもう一度だけコンパイルします。

# SyncTeX で対応箇所へ移動する

TeX のプレビューで重要なのは、ソースと PDF の対応です。この実装では、normal mode の `<CR><CR>` に SyncTeX forward search を割り当てています。

```lua
vim.keymap.set('n', '<CR><CR>', sync_from_cursor, {
  buffer = buf,
  silent = true,
  desc = 'auto_preview: SyncTeX forward search',
})
```

カーソル位置から、行番号、列番号、ファイル名を取り出します。

```lua
local pos = vim.api.nvim_win_get_cursor(win)
local line, colnr = pos[1], pos[2] + 1
```

それを `synctex view` に渡します。

```lua
vim.system({
  'synctex', 'view',
  '-i', string.format('%d:%d:%s', line, colnr, file),
  '-o', S.pdf,
}, { text = true }, function(res)
  -- 出力から page と box を読む
end)
```

SyncTeX の出力から、ページ番号と対応する box を取り出します。

```lua
local page = out:match('Page:(%d+)')
local h = tonumber(out:match('[\r\n]h:([%d.%-]+)'))
local v = tonumber(out:match('[\r\n]v:([%d.%-]+)'))
local bw = tonumber(out:match('[\r\n]W:([%d.%-]+)'))
local bh = tonumber(out:match('[\r\n]H:([%d.%-]+)'))
```

ページ番号が分かれば、そのページを表示します。box が分かれば、表示後にその位置をハイライトします。

```lua
show(tonumber(page), box and function()
  place_highlight(box)
end or nil)
```

# ハイライトは小さな画像を重ねる

`vim.ui.img` は画像を表示する API なので、ハイライトも画像として扱えます。この実装では、半透明の 1x1 PNG を作り、それを SyncTeX が返した矩形に合わせて拡大表示しています。
つまり、PDF ページ画像の上に、別の画像を overlay として重ねています。ここでも重要なのは座標変換です。SyncTeX が返すのは PDF 上の座標であり、`vim.ui.img` に渡すのはターミナル上のセル座標です。
そのため、次の情報を使って座標を変換します。

* PDF ページの画像サイズ
* 右ペイン内での画像表示位置
* セルの縦横比
* `vim.ui.img` に渡した表示幅と高さ
* SyncTeX が返した box

この変換が合っていれば、ソース上のカーソル位置に対応する PDF 上の領域を、Neovim 内でそのままハイライトできます。

# 実装して分かったこと

`vim.ui.img` は、単に「ターミナルに画像を出せる」だけの機能ではありません。行、列、幅、高さ、z-index を指定して画像を配置できるため、Neovim の UI を構成する部品として使えます。
一方で、PDF ビューアをそのまま置き換えるものでもありません。ページ移動、拡大縮小、検索、注釈、履歴といった PDF ビューアの機能を全部実装するなら、それは別の大きな話になります。
今回の用途では、必要な機能はもっと限定的です。

* 保存したら現在の PDF を見たい
* ソース位置から対応する PDF ページへ飛びたい
* 対応箇所がどこか分かればよい

この範囲なら、PDF を PNG に変換して `vim.ui.img` で表示するだけでも十分実用になります。

# まとめ

Neovim 0.13 の `vim.ui.img` を使って、TeX の Live Preview を Neovim 内に実装しました。
今回の要点は、PDF を直接扱うことではなく、PDF ページを画像に変換し、それを Neovim の UI 上に配置することです。さらに SyncTeX と組み合わせることで、ソース位置に対応する PDF ページへ移動し、該当箇所を overlay でハイライトできます。
外部 PDF ビューアを使えば、TeX の Live Preview は以前から可能でした。しかし、`vim.ui.img` によって、画像を Neovim の UI 部品として扱えるようになったことで、エディタ内に閉じたプレビュー環境をかなり自然に作れるようになりました。
まだ PDF ビューアそのものを置き換えるものではありません。それでも、執筆中に「今のソースが PDF 上でどう見えるか」を確認する用途なら、十分に使える感触です。
